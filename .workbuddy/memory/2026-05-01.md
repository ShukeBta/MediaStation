## 修复 QB 下载 Bug（torrent bencode 格式验证过严）

### 问题
错误信息：`'torrent.torrent' 不是一个有效的 torrent 文件`
qBittorrent 返回这个错误，说明传给 QB 的种子内容无效。

### 根本原因
1. **Bencode 格式验证过严**：原代码检查 `content[:4] == b"d8:"` 过于严格。合法的 `.torrent` 是 Bencode dict，只要以 `b"d"` 开头即可（`d4:`、`d6:`、`d13:` 等都是有效的）
2. **MAGNET_NEED_TRACKER 分支 `resp_text` 未定义**：`resp` 已经替换但 `resp_text` 引用了旧变量
3. **QB 响应体判断不全**：只检查 `"fail"` 关键字，没处理 QB 返回中文错误消息的情况（如 `"错误：'torrent.torrent' 不是一个有效的 torrent 文件。"`）
4. **genDlToken URL 被直接传给 QB**：M-Team API 端点不能作为 `urls` 参数给 QB

### 修复内容
- `backend/app/download/clients.py`:
  - 新增 `_is_valid_torrent()` 静态方法：只检查 `content[:1] == b"d"` 且 `len > 50`
  - 修复所有 `d8:` 硬编码改为调用 `_is_valid_torrent()`
  - QB 响应判断改为 `resp_text.lower() in ("ok.", "ok", "")` 为成功，其他全失败
  - 修复 MAGNET_NEED_TRACKER 分支 `resp_text` 未定义 Bug
  - genDlToken URL 不走 URL fallback，直接抛出有意义的错误

- `backend/app/subscribe/site_adapter.py`:
  - 新增 `_is_valid_torrent()` 静态方法
  - 修复所有 `d8:` 硬编码
  - genDlToken 响应体直接是种子二进制的情况也能处理

- `backend/app/download/service.py`:
  - `add_task` 捕获 `add_torrent` 异常，创建 `failed` 状态任务记录（不再抛 500）
  - 错误信息清晰存入 `task.message`

## 修复 M-Team genDlToken API 调用方式

### 问题
错误信息：`參數錯誤 (code=1)` — 之前一直说是"API Key 缺少下载权限"，但实际上**是调用方式错误**，与权限无关。

### 根本原因
根据 MoviePilot OpenAPI 文档（`https://api.movie-pilot.org/api/v1/openapi.json`）和 M-Team API 规范：
1. **genDlToken 参数传递错误**：之前传了 `json={"id": tid}`（JSON body），但 API 要求 **query 参数** `POST /torrent/genDlToken?id={id}`
2. **没有使用 dlV2 端点**：正确流程是 `genDlToken → 获取 credential → dlV2?credential=xxx 下载种子文件`
3. **API 路径错误**：使用了 `/api/torrent/...`（旧版），应该是 `/torrent/...`（新版 v3 API）
4. **API 基础地址不统一**：应使用 `https://test2.m-team.cc/api/v3` 作为所有 API 调用的基础

### 修复内容（`backend/app/subscribe/site_adapter.py`）
- `MTeamAdapter.API_BASE = "https://test2.m-team.cc/api/v3"` — 统一 API 基础地址
- `_download_torrent_file()` 完全重写：
  - 步骤1：`POST /torrent/genDlToken?id={tid}`（query 参数）
  - 步骤2：`GET /torrent/dlV2?credential={credential}`（下载种子）
- `search()`、`test_connection()`、`SiteResourceBrowser._browse_mteam()` 全部更新为正确路径
- 所有 `/api/torrent/` 路径改为 `/torrent/`（去掉了多余的 `/api/`）

### M-Team API 正确流程
```
1. POST https://test2.m-team.cc/api/v3/torrent/genDlToken?id={torrent_id}
   → 返回 {code:200, data: {credential: "xxx"}}

2. GET  https://test2.m-team.cc/api/v3/torrent/dlV2?credential={credential}
   → 返回 .torrent 文件二进制
```

### 参考文档
- MoviePilot OpenAPI: https://api.movie-pilot.org/api/v1/openapi.json
- M-Team API v3: https://test2.m-team.cc/api/v3/api-docs/normal

    """获取 TMDB 影视的英文标题，用于站点搜索"""
    en_client = httpx.AsyncClient(
        base_url=self.base_url,

    )
    # 请求 /movie/{id} 或 /tv/{id} 获取 original_title / original_name

#### 2. 修改订阅搜索逻辑 (`backend/app/subscribe/service.py`)
- 优先使用 `original_name`（需判断是英文）
- 通过 `get_english_title()` 获取真正的英文标题

#### 3. 新增 `original_name` 字段
- `SubscriptionCreate` / `Subscription` / `SubscriptionOut` 添加 `original_name` 字段
- 前端订阅时传递 `original_title` / `original_name`

1. `original_name`（前端保存的英文名，如果是英文）

## 修复站点连接测试 "key無效 (code=1)" 错误

### 问题
点击站点管理中的"测试连接"按钮，返回错误 `API 错误: key無效 (code=1)`，即使 API Key 和站点配置都正常。

### 根本原因
M-Team v3 API 返回 `code=1` 时：
1. 旧版逻辑将 `code=1` 视为"失败/错误"
2. 但 M-Team 新版搜索接口实际用 `code=1` 表示"成功但无数据"（搜索"test"没有匹配结果是正常的）
3. 这是 API 设计差异，不是认证失败

### 修复内容（`backend/app/subscribe/site_adapter.py`）

1. **`_parse_response()` 方法增强**：
   - 新增 `allow_code1` 参数（默认 True）
   - 当 `code=1` 且 `allow_code1=True` 时，视为"成功（无数据）"而非错误
   - 调用方可以控制是否接受 `code=1` 视为成功

2. **`test_connection()` 方法重写**：
   - 不再使用 `_parse_response()`（它默认将 code=1 视为失败）
   - 直接检查 HTTP 状态码和响应格式
   - 判断成功条件：`code in (0, 1, 200)` — 三个都算成功
   - 302 重定向才真正表示认证失败

3. **`SiteResourceBrowser._browse_mteam()` 修复**：
   - 改为检查 `code not in (0, 1, 200)` 才视为失败
   - `code=1` 现在正确返回空列表而不是错误

### 关键代码变更
```python
# 修复前（错误）
if data.get("code") == 1:
    return [], 0  # 错误地认为 code=1 是失败

# 修复后（正确）
if data.get("code") not in (0, 1, 200):
    return [], 0  # 只有非成功码才算失败
```
